## 3. Стратешки правец — Приоритети

### ПРИОРИТЕТ 1: Gymnasium испити 2016–2025 (МК) — Content Sprint

**Зошто прво ова:**
- Гимназиските наставници се biggest user segment
- Сите 10 испити (2016–2025) се достапни на ДИМ веб-сајт
- Моделот е готов — само внесување на прашања
- Веднаш им дава вредност на корисниците

**Обем на работа:** ~180 прашања (10 испити × 18 прашања просечно)
**Формат:** `MaturaExam` objects во `data/matura/index.ts` или нови фајлови по година

**Препорака за архитектура:**
```
data/matura/
  index.ts              ← агрегатор (ги увезува сите)
  gymnasium/
    2016.ts, 2017.ts, ... 2025.ts
  vocational/
    it/2022.ts, ...
    economics/2022.ts, ...
```

### ПРИОРИТЕТ 2: Август сесии за Gymnasium (2016–2025, освен 2020)

**Зошто:** Август сесијата е ист формат — само поинакви прашања. Ученикот кој padn во јуни полага во август — треба историски материјал за вежба.

**Белешки:** 2020 август нема поради COVID — ова треба да биде документирано во UI (tooltip: "Август 2020 — испитот не се одржа поради пандемијата COVID-19").

### ПРИОРИТЕТ 3: Јазични варијанти — АЛ прво, потоа ТУР

**Зошто важно:** ~25% од матурантите полагаат на АЛ јазик. ТУР е помал сегмент.

**Архитектура за јазични варијанти:**
```ts
// Пристап 1: Посебни MaturaExam објекти со language: 'al'
export const GYMNASIUM_2023_AL: MaturaExam = {
  id: 'gymnasium-2023-al',
  language: 'al',
  ...questions translated
}

// Пристап 2: questionGroupId — поврзува МК+АЛ+ТУР верзии на исто прашање
// Оваа e поскалабилна — веќе е во типот (questionGroupId?: string)
```

**Препорака: Пристап 2 (questionGroupId)** — полесно за одржување и за AI анализа ("дали ученикот збунува математиката или јазикот?")

### ПРИОРИТЕТ 4: Vocational испити (2020–2025, 6 профили)

**Зошто не прво:** Стручните испити имаат поинаков формат (понекогаш 3 части) и содржината е попрактична. Потребна е поголема аналитика на реалните DIM документи пред внесување.

**Кога:** По комплетирање на gymnasium базата.

## 4. Feature Roadmap — Надградба врз комплетна Матура база

Кога базата ќе биде комплетна (2016–2025, сите јазици), овие features стануваат возможни:

### A. Историска аналитика на испитите

```
"Теми по фреквенција 2016–2025"
→ Графикон: Деривати (23% прашања), Интеграли (18%), Тригонометрија (15%)...
→ Trend: Која тема расте/опаѓа во испитите?
```

**Вредност:** Наставникот знае ТОЧНО на што да се фокусира.

### B. Персонализирана Матура подготовка

```
Ученик решава 3 испити → AI профилира:
- "Слаб на Комплексни броеви (40% точност)"
- Генерира practice set: само прашања од слабата тема, 2016–2025
```

**Достапно веднаш** по P4 fix — `allNationalStandards` сега ги содржи secondary standards.

### C. AI-водено решавање на матурски прашања (чекор-по-чекор)

Полето `aiSolution?: string` е веќе во типот.

```
Ученик кликнува "Покажи решение" →
callGeminiProxy(прашање + getSecondaryTrackContext('gymnasium')) →
LaTeX чекор-по-чекор → cached во aiSolution
```

**Effort: МАЛО** — инфраструктурата е готова (callGeminiProxy, LaTeX renderer, caching).

### D. DoK анализа на испитите

`MaturaQuestion.dokLevel` е веќе во типот.

```
"2023 испит: DoK 1 = 35%, DoK 2 = 45%, DoK 3 = 20%"
"Трендот: испитот станува потежок по DoK секоја година?"
```

### E. Конкурентска предност: Август 2020 UI

```tsx
{exam.session === 'august' && exam.year === 2020 && (
  <Tooltip text="Овој испит не се одржа поради COVID-19 пандемијата">
    <Badge>COVID — без испит</Badge>
  </Tooltip>
)}
```

Мал детаљ, голем сигнал за доверба и прецизност.

## 5. Content Pipeline Препорака — Систематски пристап

### Фаза 1: Шаблон-базиран внес (веднаш)

Постои `docs/VOCATIONAL_CURRICULUM_EXTRACTION_TEMPLATE.md` — направи аналоген за Матура:

```markdown
# MATURA_EXAM_EXTRACTION_TEMPLATE.md

## За секое прашање внеси:
- questionNumber, questionText (точен текст)
- choices: А/Б/В/Г
- correctAnswer (провери со официјален клуч)
- topic (Деривати/Интеграли/Тригонометрија/...)
- bloomLevel (Remember/Apply/Analyze/Evaluate)
- points (1/2/3)
- dokLevel (1/2/3/4)
```

### Фаза 2: Import Pipeline (S17)

Полето `MaturaQuestion.hasImage`, `imageUrls[]`, и `importedAt` сугерираат дека S17 планирал AI-assisted import. Ова е:

```
PDF скенирани испити → Gemini Flash (OCR + LaTeX extraction) →
структурирани MaturaQuestion објекти → manual review → merge
```

**Effort: СРЕДНО** — но го забрзува внесот 10×.

### Фаза 3: Quality Assurance тестови

По секој batch внес, нови vitest тестови:
```ts
it('gymnasium 2016-2025 june: сите имаат ≥15 прашања', ...)
it('сите прашања имаат correctAnswer во choices', ...)
it('нема дупликат IDs низ сите испити', ...)
```

## 8. Едно критично техничко прашање — Data Source Strategy

**Дали треба испитите да бидат во код (`data/matura/`) или во Firestore?**

| | Код (тековно) | Firestore |
|-|--------------|-----------|
| Перформанси | ✅ instant (bundled) | 🟡 мрежа |
| Ажурирање | ❌ deployment потребен | ✅ real-time |
| Обем | 🟡 ~500KB за 10 год | ✅ неограничено |
| Offline | ✅ секогаш | 🟡 само кешираното |
| Admin editing | ❌ | ✅ CurriculumEditor style |

**Препорака:** Хибриден пристап:
- **Gymnasium 2016–2023**: во код (стабилни, bundled, offline-ready)
- **Тековна година (2024+)**: во Firestore (може да се ажурира без deployment)
- **Паметен fallback**: `firestoreService.getExams()` → fallback на `data/matura/`

Ова е веќе архитектурно можно — `MaturaExam.importedAt` и Firestore `matura_exams` collection постојат.
